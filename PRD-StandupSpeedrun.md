# Standup Speedrun — Product Requirements Document

## 1. Overview

Standup Speedrun — мультиплеерная браузерная игра для команд до 10 человек. Игроки по очереди отвечают на падающие сверху вопросы в стиле ежедневного стендапа, пока остальные поддерживают или саботируют через общий пул энергии. Игра идёт на время, побеждает тот кто набрал больше очков.

## 2. Goals & Success Metrics

- Игра запускается и полностью проходится за 3-5 минут
- До 10 игроков подключаются без регистрации
- Каждый игрок вовлечён на протяжении всей игры (нет режима "зрителя")
- Вопросы генерируются уникально для каждой сессии через ChatGPT
- Финальный экран вызывает смех и обсуждение в команде

## 3. Target Audience

**Первичная аудитория:** команды разработчиков (разработчики, тестеры, PM) на хакатонах, тимбилдингах, корпоративах.

**Характеристики:** технически грамотны, знают что такое стендап, ценят юмор связанный с разработкой.

## 4. Core Features & Functionality

### 4.1 Лобби и подключение

**Description:** Игроки заходят на общий URL, вводят имя и попадают в лобби. Ведущий (первый вошедший) задаёт контекст для генерации вопросов и запускает игру.

**User Story:** Как игрок, я хочу зайти на URL и ввести имя, чтобы сразу начать играть без регистрации.

**Acceptance Criteria:**
- [ ] Игрок заходит на `/` и видит форму ввода имени
- [ ] После ввода имени попадает в лобби и видит список подключённых игроков (real-time)
- [ ] Первый вошедший получает роль Host
- [ ] Host видит поле ввода контекста (например: "мы делаем CRM для банка") и кнопку "Сгенерировать вопросы"
- [ ] Host видит кнопку "Начать игру" — активна только если вопросы сгенерированы
- [ ] Все остальные видят статус "Ждём начала игры..."
- [ ] Максимум 10 игроков в одной сессии

**Technical Considerations:**
- WebSocket соединение устанавливается при входе в лобби
- Состояние лобби (список игроков) синхронизируется через broadcast
- Роль Host хранится в game session на сервере

---

### 4.2 Генерация вопросов через ChatGPT

**Description:** Перед стартом игры Host вводит контекст команды, система отправляет запрос в ChatGPT API и получает 30 уникальных стендап-вопросов.

**User Story:** Как Host, я хочу задать контекст нашей команды, чтобы вопросы были релевантными и смешными.

**Acceptance Criteria:**
- [ ] Host вводит контекст в текстовое поле (необязательно, есть дефолт "IT команда разработчиков")
- [ ] При нажатии "Сгенерировать вопросы" отправляется запрос к ChatGPT API
- [ ] Показывается лоадер пока идёт генерация
- [ ] Генерируется ровно 30 вопросов, сохраняются в Postgres для текущей сессии
- [ ] Host видит превью первых 5 вопросов для проверки
- [ ] Если генерация упала — показывается ошибка и кнопка "Попробовать снова"

**Prompt для ChatGPT:**
```
Ты генератор вопросов для командной игры Standup Speedrun.
Контекст команды: {context}

Сгенерируй 30 коротких смешных вопросов в стиле ежедневного стендапа.
Вопросы должны быть:
- Короткие (максимум 10 слов)
- Узнаваемые для IT команды
- Немного абсурдные и смешные
- На русском языке

Примеры: "Что сделал вчера кроме созвонов?", "Есть блокеры? Только честно.", "Когда будет готово? Примерно?"

Верни JSON массив из 30 строк.
```

**Technical Considerations:**
- Запрос к OpenAI API делается с Go backend
- Вопросы сохраняются в таблицу `questions` привязанную к `session_id`
- Таймаут запроса 15 секунд

### 4.3 Игровой цикл

**Description:** Игра идёт 3 минуты. Система последовательно назначает активного игрока — у него падает блок с вопросом. Остальные в это время используют атаки/поддержку из командного пула энергии.

**User Story:** Как игрок, я хочу видеть когда мой ход, быстро ответить на вопрос и использовать атаки на других.

**Acceptance Criteria:**
- [ ] При старте игры запускается таймер 3 минуты, виден всем
- [ ] Система рандомно выбирает первого активного игрока
- [ ] У активного игрока блок начинает падать сверху вниз за 10 секунд
- [ ] Активный игрок видит вопрос в блоке и поле ввода ответа
- [ ] Любой ответ (непустая строка) засчитывается как успешный
- [ ] После успешного ответа — блок исчезает, начисляются очки, следующий игрок становится активным
- [ ] Если блок достиг дна — штраф очков всей команде, следующий игрок становится активным
- [ ] Порядок активных игроков: round-robin по списку подключённых
- [ ] Игра заканчивается когда таймер достигает 0

**Алгоритм смены активного игрока:**
```
currentIndex = (currentIndex + 1) % len(players)
nextPlayer = players[currentIndex]
broadcast("next_turn", {activePlayer: nextPlayer.id, question: nextQuestion})
```

**Technical Considerations:**
- Game loop тикает на сервере каждые 100мс
- Позиция блока вычисляется на сервере: `position = elapsed / blockDuration * 100%`
- Сервер является источником правды для позиции блока (не клиент)
- При ответе клиент отправляет `{type: "answer", text: "..."}`, сервер валидирует

### 4.4 Командный пул энергии

**Description:** На всю команду один общий пул энергии (100 единиц). Атаки тратят энергию, пул восстанавливается со временем. Это балансирует частоту саботажа.

**User Story:** Как игрок, я хочу стратегически выбирать когда использовать атаку, зная что трачу общий ресурс.

**Acceptance Criteria:**
- [ ] Пул энергии виден всем игрокам в реальном времени (прогресс-бар)
- [ ] Пул восстанавливается на 5 единиц каждые 2 секунды
- [ ] Максимум пула: 100 единиц
- [ ] Если энергии недостаточно — кнопка атаки неактивна (серая)
- [ ] При использовании атаки энергия списывается мгновенно у всех

**Стоимость атак:**

| Атака | Стоимость | Эффект |
|-------|-----------|--------|
| 👍 Лайк | 5 | +15 очков активному игроку |
| 💣 +Блок | 20 | Добавляет лишний блок в очередь активного |
| 🔒 Заблокировать ввод | 25 | Поле ввода неактивно 4 секунды |
| 🙈 Скрыть часть вопроса | 15 | 50% текста заменяется на ░░░ на 5 сек |
| 👻 Скрыть весь вопрос | 30 | Блок полностью прозрачен на 5 сек |

**Technical Considerations:**
- Состояние пула хранится в game session на сервере
- Восстановление пула — горутина с тикером каждые 2 секунды
- Атака применяется только если `pool.energy >= attack.cost` (проверка на сервере)

### 4.5 Экраны игроков (личный + общий)

**Description:** Два типа экранов: общий (проектор) показывает всю команду и общий пул, личный (телефон/ноутбук каждого) показывает кнопки действий и поле ввода когда твой ход.

**User Story:** Как игрок, я хочу на своём экране видеть кнопки действий, а на проекторе — общую картину игры.

**Acceptance Criteria:**

**Общий экран (`/screen`):**
- [ ] Список всех игроков с их очками (обновляется real-time)
- [ ] Подсветка активного игрока
- [ ] Визуализация падающего блока с вопросом (CSS анимация)
- [ ] Прогресс-бар командного пула энергии
- [ ] Таймер игры
- [ ] Анимация при успешном ответе (зелёная вспышка + звук)
- [ ] Анимация при провале (красная вспышка + звук + "-X очков команде")
- [ ] Анимация при атаке (иконка атаки летит к блоку)

**Личный экран (`/play`):**
- [ ] Когда НЕ твой ход: кнопки атак (активные/неактивные в зависимости от пула), имя активного игрока
- [ ] Когда твой ход: поле ввода ответа с автофокусом, блок с вопросом (с эффектами атак если применены), кнопка "Ответить"
- [ ] Очки игрока и место в таблице

---

### 4.6 Финальный экран

**Description:** После окончания таймера показывается финальный экран с таблицей очков и смешными титулами для каждого игрока.

**User Story:** Как игрок, я хочу увидеть итоги и посмеяться над титулами.

**Acceptance Criteria:**
- [ ] Таблица очков от первого к последнему с анимацией появления
- [ ] Каждый игрок получает один титул
- [ ] Победитель выделен отдельно (конфетти, звук)
- [ ] Кнопка "Играть снова" (сбрасывает очки, оставляет игроков и вопросы)

**Алгоритм титулов:**

| Титул | Условие |
|-------|---------|
| 🚀 10x Developer | Больше всех успешных ответов |
| 🧱 Блокер команды | Больше всех пропущенных блоков |
| 😈 Главный саботажник | Больше всех потрачено энергии на атаки |
| 🛡️ Командный игрок | Больше всех использовал 👍 лайк |
| ☕ Проснулся к концу | Последние 30 сек ответил быстрее всех |

## 5. Technical Stack

- **Frontend:** Angular 17+ — компонентная архитектура, WebSocket через RxJS
- **Backend:** Go — WebSocket сервер (gorilla/websocket), REST для генерации вопросов
- **Database:** PostgreSQL — хранение сессий, вопросов, результатов
- **AI:** OpenAI ChatGPT API (gpt-4o-mini) — генерация вопросов
- **Real-time:** WebSocket (не SSE — нужен двусторонний канал для атак)

---

## 6. Data Model

### Session
| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| context | string | Контекст команды для генерации вопросов |
| status | enum | lobby / active / finished |
| energy_pool | int | Текущий уровень энергии (0-100) |
| started_at | timestamp | Время старта игры |
| ends_at | timestamp | started_at + 3 минуты |
| created_at | timestamp | Время создания сессии |

### Player
| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| session_id | UUID | FK → Session |
| name | string | Имя игрока |
| score | int | Текущие очки |
| answers_count | int | Успешных ответов |
| missed_count | int | Пропущенных блоков |
| energy_spent | int | Потрачено энергии на атаки |
| likes_given | int | Лайков поставлено |

### Question
| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Primary key |
| session_id | UUID | FK → Session |
| text | string | Текст вопроса |
| used | boolean | Был ли использован в игре |
| order_index | int | Порядок использования |

**Relationships:**
- Session → Players (one-to-many)
- Session → Questions (one-to-many)

## 7. UI/UX Design Principles

- **Минимализм:** никаких лишних элементов, только то что нужно прямо сейчас
- **Крупный текст:** вопросы читаемы с 2-3 метров (для проекторного экрана)
- **Цветовой язык:** зелёный = успех, красный = провал/штраф, жёлтый = атака
- **Звуки:** короткие, смешные, не раздражающие (можно заглушить)
- **Тёмная тема:** лучше читается на проекторе

---

## 8. Authentication & Security

- Авторизации нет — только имя игрока
- Сессии живут максимум 30 минут, потом удаляются из БД
- OpenAI API ключ хранится в env переменной на сервере, никогда не передаётся клиенту

## 9. WebSocket Events

**Client → Server:**
```
{ type: "join",    payload: { name: string } }
{ type: "answer",  payload: { text: string } }
{ type: "attack",  payload: { attack: "like"|"block"|"lock"|"hide_part"|"hide_all", targetId: string } }
{ type: "start" }  // только Host
```

**Server → Client (broadcast):**
```
{ type: "lobby_update",   payload: { players: Player[] } }
{ type: "game_start",     payload: { endsAt: timestamp } }
{ type: "turn",           payload: { activePlayerId: string, question: string, blockDuration: 10000 } }
{ type: "block_position", payload: { position: number } }  // каждые 100мс
{ type: "attack_applied", payload: { attack: string, effect: string } }
{ type: "energy_update",  payload: { energy: number } }
{ type: "score_update",   payload: { players: Player[] } }
{ type: "game_over",      payload: { players: Player[], titles: Record<string, string> } }
```

## 10. Development Phases

### Phase 1: MVP — Core Game Loop (первый час)
- Лобби: вход по имени, список игроков, кнопка старта
- WebSocket сервер: join, turn, answer, broadcast
- Падающий блок с вопросом (хардкод 5 вопросов без GPT)
- Смена активного игрока, базовое начисление очков
- Финальный экран с таблицей очков

**Milestone:** Можно сыграть полную игру от начала до конца

### Phase 2: Атаки и энергия (второй час)
- Командный пул энергии (логика на сервере)
- 5 атак с эффектами на клиенте
- Генерация вопросов через ChatGPT API
- Общий экран `/screen` для проектора
- Анимации и звуки
- Финальные титулы

**Milestone:** Полная игра с атаками и уникальными вопросами

## 11. Risks & Mitigation

| Risk | Impact | Mitigation |
|------|--------|------------|
| ChatGPT API медленно отвечает | Игроки ждут в лобби | Таймаут 15 сек + fallback на хардкод вопросы |
| WebSocket соединение рвётся | Игрок выпадает из игры | Автореконнект на клиенте, сервер держит состояние 30 сек |
| Синхронизация позиции блока лагает | Нечестная игра | Позиция блока на сервере — истина, клиент только рендерит |
| Все атакуют одновременно | Race condition на пуле энергии | Mutex на операции с energy_pool |
